## 1. نظام الأذونات والصلاحيات (Permissions & Access Control)

- **الصلاحيات حسب القسم:** كل قسم يجب أن يكون لديه صلاحيات خاصة تقتصر على بيانات قسمه فقط.
- **الإرث التلقائي:** عند إضافة موظف جديد إلى قسم محدد، يجب أن تنزل له الصلاحيات المُعرّفة مسبقاً لهذا القسم تلقائياً.
- **الصلاحيات المخصصة:** إذا أراد المسؤول إعطاء موظف صلاحيات وصول إضافية معينة، يمكن إضافتها له يدوياً.
- **الأدوار (Roles):** عند إنشاء حساب مستخدم للموظف، يتم ربطه بأدوار (Roles) وهي عبارة عن صلاحيات محددة. يوجد خيار "ربط حساب مستخدم موجود" يظهر قائمة بأسماء المستخدمين الحاليين (مثل: علاء الدهبلي وغيره) لربط الموظف بحساب موجود مسبقاً.

## 2. نظام التذاكر (Tickets / Support Requests)

### 2.1 الواجهة العامة (إنشاء التذكرة)
يجب أن تكون واجهة واحدة بسيطة تحتوي على الحقول التالية:

| الحقل | التفاصيل |
|-------|----------|
| القسم المستلم | القسم الذي سترسل له التذكرة |
| قسم إضافي | خيار إضافة قسم ثانٍ إذا كانت التذكرة موجهة لقسمين في نفس الوقت |
| الموظف المستلم | اختيار موظف من القسم/الأقسام المختارة (اختياري) |
| التاريخ والوقت | تاريخ ووقت إنشاء/إرسال التذكرة |
| رقم التذكرة | يُولّد تلقائياً حسب اسم الشركة، ويبدأ العدّ عند إنشاء حساب الشركة |
| الموضوع | عنوان التذكرة |
| الوصف | تفاصيل الطلب |
| المرفق | إمكانية رفع مرفقات |
| مستوى الأهمية | `High` &vert; `Mid` &vert; `Low` |
| الحالة | `تسليم` &vert; `مسودة` &vert; `إلغاء` |

### 2.2 حالة الإجازة المرضية
- إذا كانت التذكرة من نوع "إجازة مرضية"، يجب أن يُطلب من الموظف إرفاق مرفق من **منصة صحتي**.
- يتم التحقق من صحة الإجازة المرضية من خلال **المدير المباشر** و**موظف الموارد البشرية (HR)**.

### 2.3 تتبع حالة التذكرة
- عند استعراض التذكرة لأول مرة، تظهر حالتها بـ **"في حالة الانتظار"**.
- إذا فتحها القسم المعين (دون تحديد موظف محدد)، يظهر تحديث:  
  **"تم استعراضها بواسطة [اسم الموظف]"**.
- إذا كانت موجهة لشخص محدد، تكون الحالة مبنية على الإجراء الذي يقوم به المستلم.

### 2.4 واجهة المستلم (داشبورد التذاكر)
- تظهر التذكرة لدى المستلم لاستعراض محتواها.
- يمكن للمستلم:
  - كتابة تعليق على التذكرة.
  - إضافة مرفق جديد.
- أزرار تحديث الحالة في داشبورد التذاكر الخاصة بالموظف:
  - `معاينة`
  - `جاري المعالجة`
  - `محلولة`

### 2.5 ملاحظة جوهرية
- التذاكر **واجهة بسيطة فقط**، و**لا تحتاج إلى موافقات رسمية** أو سلسلة اعتمادات.
- التذاكر تشبه إرسال **إيميل** — ترسل لشخص أو قسم لإعلامه بأمر ما، دون سير عمل ثابت.

## 3. نظام المهام (Tasks)

### 3.1 الفرق الجوهري بين التذاكر والمهام
- **التذاكر:** واجهة بسيطة، تشبه إرسال إيميل، لا تحتاج إلى موافقات أو سير عمل ثابت. ترسل لقسم أو شخص لإعلامه بأمر ما.
- **المهام:** لها **سير عمل محدد وثابت** من قسم إلى آخر، ضمن خطة سير واضحة ذات بداية ونهاية محددتين. المهام مترابطة بشكل كامل.

### 3.2 إنشاء المهمة
- يتم إنشاء المهمة لقسم معين (مثلاً: قسم المناقصات، قسم التندر، قسم المبيعات).
- يمكن إنشاء المهمة لنفس القسم (القسم الذي ينشئها) أو لقسم آخر.
- **لا يوجد** خيار "إضافة قسم آخر" كما في التذاكر. يتم اختيار قسم واحد فقط.
- عند إنشاء مهمة، يتم تحديد:
  - نوع المهمة (مثلاً: منافسة جديدة)
  - رقم المنافسة (إن وجد)
  - اسم المنافسة
  - القسم المستلم
- الواجهات **موحدة** لكنها تختلف من قسم إلى قسم حسب الحقول المطلوبة لكل قسم.

### 3.3 سير عمل المهام (مستخرج من الـ Flowchart المرفق)

```
واجهة إنشاء المهمة
    ↓
إدخال: القسم، المرفق، العنوان، التاريخ
    ↓
مراجعة مبدئية؟
    ├─ نعم → إحالة لقسم المبيعات المحدد بالواجهة
    └─ لا  → إغلاق/رفض المهمة
    ↓
مراجعة الجدوى؟
    ├─ نعم → موافقة فريق المبيعات → اعتماد مدير المبيعات
    └─ لا  → إغلاق/رفض المهمة
    ↓
تكلفة المناقصة حسب الإدخال بالواجهة
    ├─ شراء كراسة:
    │     المناقصات: طلب دفع رسوم الكراسة
    │     المالية: سداد الرسوم وإرفاق الفاتورة
    └─ محلية:
    │     المناقصات: طلب وتحميل الكراسة مباشرة بدون مرور بالمالية
    ↓
المناقصات: تحميز ملف المناقصة
    ↓
المبيعات: التسعير والمستندات الفنية
    ↓
التنظيمية: توفير التراخيص والامتثال
    ↓
المراجعة النهائية؟
    ├─ معتمد → المناقصات: تسليم المناقصة بالبوابة
    └─ تعديل الوثائق → (إعادة للتصحيح)
    ↓
المناقصات: استلام العرض وتوجيهه
    ├─ المبيعات: المناقشة والتفاوض المالي/الفني
    └─ التنظيمية: المناقشة والتفاوض التنظيمي
    ↓
المناقصات: مراجعة مخرجات المناقشة
    ↓
هل صدرت ترسية مبدئية؟
    ├─ نعم:
    │     إعلان الترسية المبدئية
    │     المناقصات: مراجعة نتيجة الترسية
    │     المبيعات: تقييم الترسية
    │     يوجد مبرر للاعتراض؟
    │         ├─ نعم، يوجد مبرر:
    │         │   المناقصات: فتح طلب الاعتراض
    │         │   المالية: سداد رسوم وضمان الاعتراض
    │         │   المناقصات: استلام تأكيد السداد
    │         │   المبيعات: بناء المذكرة الفنية للاعتراض
    │         │   المناقصات: تسليم الاعتراض رسمياً
    │         │   نتيجة الاعتراض من الجهة (قبول/رفض)
    │         └─ لا، قبول النتيجة → الترسية النهائية
    └─ لا → إغلاق ورفض المهمة
```

## 4. نظام الإجازات (Vacation / Leave Management)

### 4.1 قواعد حساب رصيد الإجازات
- **الموظف الذي تتجاوز مدة خدمته 5 سنوات:** يستحق **30 يوم** إجازة سنوية.
- **الموظف الذي تقل مدة خدمته عن 5 سنوات:** يستحق **22 يوم** إجازة سنوية.
- **الإضافة التلقائية:** يتم إضافة رصيد الإجازات تلقائياً إلى النظام في **بداية كل سنة ميلادية**.

### 4.2 واجهة إضافة طلب إجازة (الحقول)

| الحقل | التفاصيل |
|-------|----------|
| اسم الموظف | يظهر تلقائياً عند فتح الطلب من حساب الموظف (مباشرة) |
| نوع الإجازة | إجازة سنوية / إجازة مرضية / إجازة طارئة / إلخ |
| رصيد الإجازة | يظهر **الرصيد المسموح** و**الرصيد المتبقي** الحقيقي للموظف (لا يظهر رصيد عام مثل "رصيد 2026") |
| تاريخ البدء | تاريخ بداية الإجازة |
| تاريخ الانتهاء | تاريخ نهاية الإجازة |
| الأيام المطلوبة | يُحسب **تلقائياً** من الفرق بين تاريخي البدء والانتهاء، **باستثناء أيام العطل (الجمعة والسبت)** |
| السبب | يظهر فقط إذا كانت الإجازة **غير سنوية** (مثلاً: إجازة طارئة). إذا كانت سنوية، لا يحتاج إلى سبب |
| المرفق | إجباري فقط إذا كانت الإجازة **مرضية**، وإلا فهو اختياري |
| طلب تأشيرة خروج وعودة | حقل إضافي: **"هل تحتاج إلى طلب تأشيرة خروج وعودة؟"** (نعم/لا) |
| حالة الطلب | معتمد / بانتظار الاعتماد / إلخ |
| تاريخ الاعتماد | يظهر تلقائياً عند الاعتماد |

### 4.3 مسار اعتماد طلب الإجازة

```
1. الموظف يقدم طلب الإجازة (يضغط "إرسال").
        ↓
2. يذهب الطلب مباشرة إلى المدير المباشر.
        ↓
3. المدير المباشر (موافقة / رفض)
   ├─ رفض → يعود الطلب إلى الموظف.
   └─ موافقة → تحويل إلى المسؤول عن الإجازات في الفرع/الشركة.
        ↓
4. المسؤول عن الإجازات (موافقة / رفض)
   ├─ رفض → يعود الطلب إلى الموظف.
   └─ موافقة → تحويل إلى مدير HR.
        ↓
5. مدير الموارد البشرية → الاعتماد النهائي.
```

### 4.4 الإجازة المرضية — قواعد خاصة
- إذا اختار الموظف **إجازة مرضية**، يكون **المرفق إجبارياً** (وليس اختيارياً).
- المرفق يجب أن يكون من **منصة صحتي**.
- التحقق من الإجازة المرضية يتم من خلال **المدير المباشر** و**موظف HR**.

### 4.5 التأمين الطبي — قواعد خاصة
- إذا اختار الموظف **نوعاً من التأمين الطبي**، يُجبر على رفع **البدل** (المرفق المالي).
- إذا اختار الموظف **جهازاً طبياً** (مثل سيارة إسعاف)، يجب أن يكون مُجبراً على ذلك.
- نوع سيارة الإسعاف مكتوب أعلاه: "استلم سيارة الإسعاف، وخذها إلى المركز الصحي."

## 5. نظام رحلات العمل (Business Trips)

### 5.1 مسار اعتماد طلب رحلة العمل

```
1. الموظف يرفع طلب رحلة عمل.
        ↓
2. تحويل الطلب إلى المدير المباشر.
   ⚠️ لا يُسمح بتقديم طلب رحلة عمل جديد طالما أن الطلب السابق ما زال بانتظار قرار المدير المباشر.
        ↓
3. المدير المباشر (موافقة / رفض)
   ├─ رفض → يعود الطلب إلى الموظف.
   └─ موافقة → تحويل إلى HR.
        ↓
4. موظف HR (موافقة / رفض)
   ├─ رفض → يجب تسجيل سبب الرفض.
   └─ موافقة → تحويل إلى مدير HR.
        ↓
5. مدير HR → اعتماد.
        ↓
6. الإدارة المالية → الاعتماد النهائي.
```

### 5.2 التتبع
- يجب توفير تتبع حالة رحلة العمل في جميع مراحلها.

## 6. نظام العمل الإضافي (Overtime)

### 6.1 قواعد احتساب العمل الإضافي
- **من الأحد إلى الخميس:** يُحتسب بـ **ساعة ونصف** (1.5×) عن كل ساعة إضافية.
- **في أيام العطل والإجازات الرسمية:** يُحتسب بـ **ساعتين** (2×) عن كل ساعة إضافية.

### 6.2 طريقة احتساب قيمة الساعة الإضافية

```
قيمة ساعة العمل الإضافي = الراتب ÷ 30 يوم ÷ 9 ساعات عمل يومية
```

### 6.3 التتبع
- يجب توفير تتبع حالة طلبات العمل الإضافي.

## 7. نظام السلف المالية (Salary Advances)

### 7.1 قواعد الخصم من الراتب
- بعد اعتماد السلفة من الإدارة المالية، يتم **خصمها تلقائياً** من راتب الموظف.
- **إذا تم اعتماد السلفة قبل يوم 20 من الشهر:** تُخصم من راتب **نفس الشهر**.
- **إذا تم اعتماد السلفة بعد يوم 20 من الشهر:** تُخصم من راتب **الشهر التالي**.

### 7.2 التقارير المطلوبة
- تقرير يعرض جميع السلف التي **تم اعتمادها** من الإدارة المالية.
- تقرير يعرض جميع السلف التي **لم يتم اعتمادها** بعد (الموقفة عند المالية).

### 7.3 التتبع
- يجب توفير تتبع حالة طلب السلفة في جميع مراحلها.

## 11. بيانات الموظف (Employee Profile Data)

### 11.1 إضافة موظف جديد — ترتيب الحقول والأقسام

#### أ) البيانات الشخصية (الجزء الأول)
| الحقل | التفاصيل |
|-------|----------|
| الجنسية | عند اختيار "يمني" (أو أي جنسية)، يتم تطبيقها مباشرة |
| الاسم الأول | الاسم الأول بالعربية |
| اسم العائلة | اسم العائلة بالعربية |
| الاسم باللغة الإنجليزية | يظهر **أسفل** الاسم الأول واسم العائلة |
| البريد الإلكتروني | بريد الموظف |
| الهاتف | رقم هاتف الموظف |
| رقم الهوية | رقم الهوية الوطنية |
| تاريخ الميلاد | تاريخ ميلاد الموظف |
| الجنس | ذكر / أنثى |

#### ب) البيانات الوظيفية
| الحقل | التفاصيل |
|-------|----------|
| الفرع | اختيار الفرع |
| القسم | اختيار القسم |
| المنصب / المسمى الوظيفي | وظيفة الموظف |
| الرقم الوظيفي | الرقم الوظيفي للموظف |
| تاريخ التعيين | تاريخ بدء العمل |
| حالة التوظيف | حالة الموظف الحالية |
| المشرف المباشر | المدير المباشر للموظف |

#### ج) المستندات والهوية (مع مرفق لكل حقل)
| الحقل | المرفق المرتبط |
|-------|----------------|
| رقم جواز السفر | مرفق صورة جواز السفر (يجب أن يكون **جنب** أو **تحت** الحقل) |
| تاريخ إصدار جواز السفر | — |
| تاريخ انتهاء جواز السفر | — |
| مهنة الموظف في الهوية | مرفق صورة الهوية (يجب أن يكون **جنب** أو **تحت** الحقل) |
| رقم التأمين الاجتماعي | — |

**ملاحظة تصميمية:** يجب أن يكون **كل حقل** جنبه أو تحته **مرفقه الخاص** (صورة/ملف). مثلاً:
- رقم الجواز → ثم تاريخ الإصدار → ثم تاريخ الانتهاء → ثم **مرفق صورة الجواز** (تحته أو جنبه).
- مهنة الموظف في الهوية → **مرفق صورة الهوية** (جنبها).
- رقم التأمين الاجتماعي → يظهر منفصلاً.

#### د) البيانات المالية
| الحقل | التفاصيل |
|-------|----------|
| الراتب الأساسي | الراتب الأساسي للموظف |
| بدل المواصلات | بدل النقل |
| بدل السكن | بدل السكن |
| بدل الاتصال | بدل الاتصالات |
| بدل السفر | بدل السفر (إن وجد) |
| بدلات أخرى | أي بدلات إضافية |
| رقم الآيبان (IBAN) | عند إدخاله، يظهر **اسم البنك تلقائياً** |
| رقم الحساب | رقم الحساب البنكي |

#### هـ) معلومات الاتصال والطوارئ
| الحقل | التفاصيل |
|-------|----------|
| هاتف الاتصال | رقم هاتف إضافي |
| العنوان | عنوان السكن |
| اسم شخص في حالة الطوارئ | اسم شخص للاتصال في حالات الطوارئ |
| هاتف الطوارئ | رقم هاتف الشخص في حالات الطوارئ |

#### و) حساب المستخدم (User Account)
| الحقل | التفاصيل |
|-------|----------|
| إنشاء حساب | إنشاء حساب مستخدم للموظف |
| كلمة السر | كلمة المرور |
| الأدوار (Roles) | صلاحيات المستخدم |
| ربط حساب مستخدم موجود | خيار اختيار مستخدم موجود مسبقاً من قائمة تظهر أسماء المستخدمين الحاليين |

### 11.2 الموظف الأجنبي — الحقول الشرطية
- إذا كانت جنسية الموظف **أجنبية** (مثل: يمني، مصري، إلخ):
  - يظهر حقل إضافي: **المهنة في الإقامة**.
  - يُعرض أسفل حقل **المسمى الوظيفي**.
- إذا كان الموظف **غير أجنبي** (سعودي/محلي)، لا يظهر هذا الحقل.

### 11.3 المرفقات العامة
- إمكانية رفع مرفقات متعددة.
- يجب كتابة **اسم أو وصف** لكل مرفق على حدة.

### 11.4 الملف التعريفي
- إمكانية إرفاق **صورة شخصية** للملف التعريفي للموظف.

## 12. مسير الرواتب (Payroll Processing)

### 12.1 ما يتم احتسابه قبل إصدار المسير
- التأخير في الحضور.
- التأخير أو النقص في الانصراف.
- أي خصومات مرتبطة بالدوام (الحضور والانصراف).

### 12.2 خيارات تطبيق الخصومات (قبل إصدار المسير)
يقوم نظام HR بعرض السؤال: **"هل تريد تطبيق الخصومات؟"**

| الخيار | الوصف |
|--------|-------|
| تطبيق على جميع الموظفين | يتم احتساب التأخير والانصراف وتطبيق الخصومات على الجميع. |
| تطبيق على موظف محدد | اختيار موظف معيّن وتطبيق الخصومات عليه فقط. |
| تطبيق على عدة موظفين محددين | اختيار مجموعة موظفين وتطبيق الخصومات عليهم فقط. |
| عدم تطبيق الخصومات | إصدار المسير بدون أي خصومات، واعتماد **إجمالي الراتب الكامل** للجميع. |

## 13. التقارير المطلوبة (Reports)

### 13.1 تقارير رحلات العمل
- تقرير يعرض جميع طلبات رحلات العمل التي **تم اعتمادها** من الإدارة المالية.
- تقرير يعرض جميع طلبات رحلات العمل التي **بانتظار اعتماد** الإدارة المالية.

### 13.2 تقارير السلف المالية
- تقرير بالسلف **المعتمدة**.
- تقرير بالسلف **غير المعتمدة** (الموقفة عند المالية).

### 13.3 التقارير المالية العامة (جديد)
- **تقرير مالي** يعرض جميع الطلبات التي **تم اعتمادها من الإدارة المالية** (يشمل: السلف، الإجازات، رحلات العمل، أو أي طلب آخر يمر على المالية).
- **تقرير مالي** يعرض جميع الطلبات التي **لم يتم اعتمادها** من الإدارة المالية (الموقفة عند المالية).

## 14. سلاسل الموافقات والصلاحيات العامة (Approval Chains)

- يجب أن يكون النظام يدعم **سلاسل موافقات وصلاحيات** لجميع الأنظمة الفرعية:
  - رحلات العمل
  - العمل الإضافي
  - السلف
  - طلب الخطابات
  - العهد
  - المصاريف
- يجب أن تكون واضحة وقابلة للتخصيص حسب نوع الطلب.
- **المدير المباشر** هو أول مرحلة في معظم السلاسل.
- **الموارد البشرية (HR)** هي المرحلة الثانية.
- **مدير الموارد البشرية** هو المرحلة الثالثة (الاعتماد النهائي في بعض الأنظمة).
- **الإدارة المالية** هي المرحلة الأخيرة في الأنظمة التي تحتاج اعتماد مالي.
- إذا تم الرفض في أي مرحلة، يجب **تسجيل سبب الرفض**.

## 19. ملاحظات تقنية وتصميمية عامة

### 19.1 التذاكر
- واجهة بسيطة، لا تحتاج إلى موافقات رسمية أو سلاسل اعتماد.
- التركيز على السرعة والمرونة في التواصل بين الأقسام.
- تشبه إرسال إيميل — ترسل لشخص أو قسم لإعلامه بأمر ما.

### 19.2 المهام
- تحتاج إلى **سير عمل ثابت وواضح** (Workflow) من قسم إلى آخر.
- يجب أن يكون هناك **بداية ونهاية واضحة** لكل مهمة.
- المهام **مترابطة بشكل كامل** — كل مرحلة تعتمد على التي قبلها.
- يُرجى الالتزام بالـ Flowchart المرفق كمرجع تصميمي أساسي.
- لا يوجد خيار "إضافة قسم آخر" كما في التذاكر. يتم اختيار قسم واحد فقط.

### 19.3 الأرقام التلقائية
- رقم التذكرة: يبدأ حسب اسم الشركة عند إنشاء حساب الشركة.
- رقم طلب الإجازة: يُولّد تلقائياً.
- رقم طلب رحلة العمل: يُولّد تلقائياً.

### 19.4 التكامل مع منصات خارجية
- **منصة صحتي:** يجب التحقق من الإجازات المرضية المرفقة من هذه المنصة.
- **التحقق البنكي:** عند إدخال IBAN، يتم التعرف على اسم البنك تلقائياً (ربما عبر API بنكي أو خدمة تحقق محلية).

### 19.5 التصميم والواجهات
- **واجهة إضافة موظف:** يجب أن تكون الحقول مرتبة بالترتيب المذكور أعلاه (بيانات شخصية → وظيفية → مستندات → مالية → اتصال → حساب مستخدم).
- **المرفقات:** يجب أن تكون **جنب** أو **تحت** كل حقل مستند (جواز، هوية، إلخ)، وليس في قسم منفصل.
- **الإجازات:** يجب أن يظهر اسم الموظف تلقائياً، والأيام المطلوبة تُحسب تلقائياً باستثناء الجمعة والسبت.
- **الرصيد:** يظهر الرصيد الحقيقي للموظف (المسموح والمتبقي)، لا رصيد عام.

## 15. إعادة تصميم تجربة الاستخدام (UX Redesign)

### 19.1 تقليل الخطوات والتنقل
- يجب إعادة تصميم تجربة الاستخدام بحيث يتم **تقليل عدد الخطوات والتنقل** داخل النظام إلى الحد الأدنى.
- أي كيان يحتوي على عدة تبويبات (Tabs) يجب إعادة تنظيمه بحيث يتم **دمج المحتوى داخل واجهة واحدة موحدة**.
- بدلاً من توزيع المعلومات على تبويبات متعددة، يتم عرض كل المعلومات بشكل منظم داخل **صفحة واحدة أو عرض موحد**.
- الهدف: تقليل التنقل بين الشاشات وجعل الوصول للبيانات **أسرع وأسهل**.

### 19.2 مثال تطبيقي: كيان الموظف
- **الحالي:** بيانات الموظف موزعة على عدة تبويبات.
- **المطلوب:** عرض كل معلومات الموظف (البيانات الشخصية، الوظيفية، المستندات، المالية، الاتصال، حساب المستخدم) في **واجهة واحدة موحدة**.
- يجب أن تكون الواجهة منظمة ويسهل قراءتها واستخدامها، مع تقليل التعقيد إلى الحد الأدنى.
